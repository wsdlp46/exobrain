# 知识库搭建与运维指南

> 覆盖 Workspace 级大知识库（跨项目共享）和项目级小知识库（单项目专用）的完整搭建、日常运转与定期治理规范。本文是外脑（Exobrain）帮助新人从零搭建工作知识库的核心参考。

---

## 一、双层知识库体系

### 1.1 两层分工

```
大知识库（Workspace 级，跨项目共享）
  工作区根/
  ├── 01通用/          ← 模板、检查清单、设计规范、方法论
  ├── 03知识库/        ← 案例库、政策法规、竞品分析、技术学习
  └── 00主页.md        ← 工作区入口索引

        │ 引用（项目引用大知识库的模板、案例、规范）
        ▼

项目知识库（Project 级，单项目专用）
  项目名/
  ├── 00-项目总览.md   ← 仪表盘（一句话+关键事实+当前阶段）
  ├── _index.md         ← 文件全量索引 + 最近更新日志
  ├── 01-项目输入/     ← 本项目的会议时间线、关键决策
  ├── 02-工作产物/     ← 本项目的核心产出物
  ├── ...
  ├── _source/          ← 二进制文件统一归档 + 转换产物
  └── .workbuddy/memory/ ← AI 记忆系统
```

| 层级 | 内容 | 共享范围 | 生命周期 |
|------|------|---------|---------|
| **大知识库** | 通用模板、设计规范、案例库、政策法规、竞品分析 | 所有项目共享 | 持续累积，不随项目结束而归档 |
| **项目知识库** | 本项目输入、产物、评审、对外材料 | 本项目内 | 随项目启动创建，结束后可沉淀案例到大知识库 |

### 1.2 大知识库（Workspace 级）目录结构

```
01通用/
├── _index.md             # 统一入口索引
├── 工作流规范.md          # 跨项目通用操作规则 + AI 行为契约
├── 知识库搭建指南.md      # 搭建与治理规范（即本文档）
├── 模板/                  # 可复用的文档模板
├── 检查清单/              # 各类评审和质量检查清单
├── 设计规范/              # 设计标准（不可随意修改的参数）
├── 设计库/                # 可复用的设计组件和跨项目约定
└── 方法论/                # 模式笔记、踩坑笔记

03知识库/
├── _index.md              # 目录索引 + 已有资料清单
├── 工作案例库/             # 按领域分类的历史项目案例
├── 政策法规/               # 行业标准、政策文件
├── 竞品分析/               # 竞品调研文档
├── 技术学习/               # 技术资料、学习笔记
├── 行业研究/               # 行业洞察、趋势分析
└── 读书笔记/               # 书籍阅读笔记
```

### 1.3 项目知识库（Project 级）目录结构

```
项目名/
├── 00-项目总览.md           # 项目仪表盘（一句话+关键事实+当前阶段）
├── _index.md                # 项目文件全量索引 + 最近更新日志
├── 01-项目输入/             # 外部输入（合同/需求/会议记录）
│   ├── _index.md
│   ├── 会议纪要_时间线.md    # 按时间记录每次外部输入及影响
│   ├── 会议纪要/             # 会议原始记录
│   ├── 关键决策.md           # 跨阶段提炼的关键设计约束（#D001起）
├── 02-工作产物/             # 项目核心产出物（按需创建子目录）
├── _source/                 # 所有二进制文件统一归档
│   └── *_转换.md            # 二进制→MD 转换产物，AI 优先读取
├── .workbuddy/
│   └── memory/              # AI 记忆系统
│       ├── MEMORY.md        # 长期约定
│       └── YYYY-MM-DD.md    # 每日工作日志
```

## 二、搭结构：从零开始

### 2.1 判断工作模式

启动前，先扫描用户提供的材料，判断适合的目录模板（详见 `directory-templates.md` 中的 4 种模板）：

| 判断信号 | 推荐模板 |
|---------|---------|
| 文件按项目文件夹组织 + 有交付物 | 项目制 |
| 按业务线/功能模块组织 + 时间线主轴 | 持续运营 |
| 两者兼有 | 混合模式 |
| 以个人笔记/学习材料为主 | 个人知识库 |

### 2.2 新项目启动 Checklist

创建项目知识库后，按以下清单完成初始化：

- [ ] 创建项目目录骨架（01-项目输入/ + 02-工作产物/ + _source/ + .workbuddy/memory/）
- [ ] 编写 `00-项目总览.md`（仪表盘：快速入口、项目简介、当前阶段、关键事实）
- [ ] 编写各目录 `_index.md`（文件清单 + 目录说明）
- [ ] 初始化 `01-项目输入/关键决策.md`（空文件，标注 #D001 起编）
- [ ] 初始化 `01-项目输入/会议纪要_时间线.md`（空文件，标注首次记录准备）
- [ ] 建立 `.workbuddy/memory/MEMORY.md`，记录设计约定：
  - 颜色规范（品牌主色、组件色值）
  - 单位体系和换算规则
  - 常用模板和文件路径约定
- [ ] 首次入库测试：放一份文件，验证五步法跑通
- [ ] 提示用户："结构已就绪，接下来可以逐步入库材料，说'入库这个文件'即可开始"

## 三、养知识：日常运维

### 3.1 材料入库五步法

> **驱动 Skill**：`project-intake`

#### 步骤 1：判断类型

| 类型 | 识别信号 | 动作 |
|------|---------|------|
| PDF/DOCX/XLSX/PPTX | 扩展名 | 归档 `_source/` → 执行转换 |
| 会议记录 | 口述/粘贴，或文件名含"会议""沟通""纪要" | 提取时间/来源/内容 |
| 需求/合同类 | 文件名含"合同""需求""招标""SOW" | 存放 `01-项目输入/`，标记为原始输入 |

#### 步骤 2：二进制转换

所有二进制文件统一归档至 `_source/`，转换产物 `{原文件名}_转换.md` 与源文件同放 `_source/` 中。

| 源格式 | 转换工具 | 输出规范 |
|--------|---------|---------|
| PDF | `pypdf` 提取文本 | 逐页标注 `## 第N页`，保留原文结构 |
| DOCX | `python-docx` 提取段落 | 保留标题层级（H1→H2），正文为段落 |
| XLSX | `pandas` 提取表格 | 按 Sheet 分节，表格转 Markdown table |
| PPTX | `python-pptx` 提取文本 | 按页组织，每页标注标题和文本框内容 |

**转换优先级判断**：

| 优先级 | 判断标准 | 示例 |
|--------|---------|------|
| 必转 | 工作流中会被反复引用的核心文档 | 合同、招标文件、技术标准 |
| 可选 | 仅归档查看、不频繁读取的 | 旧版本文档、历史快照 |
| 跳过 | 内容已在其他 MD 中充分覆盖 | 重复备份、旧版草稿 |

**AI 读取规则**：优先读 `_转换.md` 产物；信息不完整时回退原始文件。

#### 步骤 3：会议时间线更新

追加到 `01-项目输入/会议纪要_时间线.md`，按时间倒序排列：

```markdown
## YYYY-MM-DD | [参会人员] [会议主题]

**核心内容**：（3句话概括）

**对项目的影响**：
- 影响1：具体描述
- 影响2：具体描述

**关联**：[[原始记录路径]]
```

#### 步骤 4：关键决策提炼

扫描决策信号词（"确定""决定""不做了""砍掉""甲方说""客户要求"），草拟条目：

```markdown
### #DXXX | YYYY-MM-DD | [决策标题]

**依据**：（为什么要这样决定）
**影响范围**：（功能范围/技术选型/交互模式/报价/排期）
**关联输入**：[[会议纪要_时间线#YYYY-MM-DD]]
**状态**：待确认
```

**关键规则**：
- 编号 `#D001` 起全局自增，读取已有最大编号再 +1
- 新决策默认"待确认"，禁止自动写"已确认"——必须等用户审阅
- 每条决策"关联输入"指向对应会议时间线条目
- 后续被推翻的决策标注 `后续决策：#D005`

#### 步骤 5：更新索引

- 更新项目 `_index.md` 文件清单和"最近更新"
- 更新对应子目录 `_index.md`
- 如涉及知识库文件，同步更新大知识库索引

**入库完成输出格式**：

```markdown
[项目名] 入库完成

文件：
- [文件名] → [存放位置]

提炼：
- 会议时间线新增 X 条
- 关键决策草拟 #Dxxx（待确认）

影响提醒：
- [如有影响现有需求的条目，在此标注]
```

### 3.2 _index.md 维护规则

每个目录的 `_index.md` 是 AI 进入该目录的"第一眼"，需保持更新：

**子目录 `_index.md` 模板**：

```markdown
# 目录名 — 目录索引

> 一句话描述

## 文件清单
| 文件 | 说明 | 最后更新 |
|------|------|----------|
| `xxx.md` | 说明 | YYYY-MM-DD |
```

**顶层 `_index.md` 须包含**：
```markdown
## 目录导航
| 目录 | 文件数 | 说明 |
|------|--------|------|

## 最近更新
- YYYY-MM-DD | 变动内容 | 关联文件
```

**维护时机**：新增或删除文件后更新；项目状态变更后同步更新项目总览。

## 四、管知识：定期治理

### 4.1 大知识库各子目录维护

| 子目录 | 新增内容来源 | 更新时机 | 过期处理 |
|--------|------------|---------|---------|
| 模板/ | 项目中发现可复用的文档结构 | 项目复盘或阶段性沉淀 | 旧版标记 `~~已过期~~`，指向新版 |
| 检查清单/ | 评审中发现的漏检项 | 每次评审后补充 | 不再适用的条目删除 |
| 设计规范/ | 新类型项目的设计标准 | 项目启动时更新参数 | 版本化管理 |
| 设计库/ | 项目中沉淀的组件模式 | 成果完成后 | 保持精选，不膨胀 |
| 方法论/ | 踩坑经验、可复用模式 | 积累 ≥3 个相似案例后提炼 | 错误经验标注上下文限制 |
| 工作案例库/ | 完成的完整项目案例 | 项目阶段性完成或出具可复用结论 | 引用的新启动项目检视时效性 |
| 政策法规/ | 行业标准、政策文件 | 二进制文件需转换后入库 | 定期查版本更新 |
| 竞品分析/ | 调研完成后入库 | 随时 | 标注调研时间，超 1 年标记时效性警告 |

### 4.2 案例库沉淀标准

**沉淀触发条件**（满足任一即可）：
- 项目阶段性完成，产出了可复用的设计模式或方案结构
- 项目中踩坑后提炼出值得记录的教训
- 复盘后发现某个决策逻辑可供后续项目参考

**案例格式**（至少包含 4 项）：
1. 背景（什么项目/场景触发）
2. 关键决策（选了 A 而非 B，为什么）
3. 踩坑记录（怎么发现的，怎么解决的）
4. 复用信号（什么情况下可以再次参考）

**积累规则**：一个领域积累 ≥3 个案例后，提炼共性为方法论笔记。

### 4.3 定期清理

建议每季度或按需执行一次：

1. **有效期检查**：政策法规是否有版本更新，标准是否过期
2. **转换时效检查**：二进制文件是否更新，转换产物是否需要重新生成
3. **案例库盘点**：是否有已完成项目未沉淀案例
4. **膨胀控制**：过期的 MD 标记 `~~已过期~~`，标注替代版本链接；确认无价值的直接删除
5. **交叉引用检查**：已过期内容的引用是否已更新指向

## 五、AI 协作规则

### 5.1 基础行为契约

外脑执行以下任务时自动遵守：

1. **入库前确认归属**：收到新文件时，先确认属于哪个项目/业务域，不明确则询问
2. **会议后提炼决策**：每次会议记录入库后，扫描关键决策信号并草拟 #Dxxx
3. **引用使用编号**：引用关键决策时使用 `#D001` 编号格式，确保全文可检索
4. **查阅二进制材料时优先读转换产物**：不直接读 PDF/DOCX/XLSX/PPTX
5. **编辑后更新索引**：新增、删除或移动文件后，更新相关 `_index.md`
6. **阶段性完成时提醒复盘**：项目里程碑完成后，主动提醒是否执行复盘
7. **踩坑后提议入库**：遇到值得记录的问题时，主动询问是否记入案例库

### 5.2 文档生成纪律（10 条）

撰写或编辑任何文档时遵守：

#### 生成时

1. **完整性校验**：源文档有 N 项，产出不得少于 N 项；有合并或省略必须显式说明原因
2. **示例不等于定义**：源文档中"如""举例""假设"等示例数值，产出中还原为规则描述，不得将示例写死为定义
3. **最小信息密度**：每个功能点至少保留"做什么 + 怎么用 / 为什么"两层信息
4. **角色必带边界**：每个角色首次出现时，用一句话交代"能做什么 + 不能做什么"
5. **流程闭环校验**：每个操作流程检查"触发条件 → 操作步骤 → 执行结果"是否完整

#### 编辑时

6. **编号一致性**：插入、删除、移动编号项后，全局刷新后续编号和交叉引用
7. **编辑后自查**：检查重复段落、断链引用、编号不连续，发现即修

#### 输出时

8. **中文编码处理**：解压含中文文件名的 zip 时优先使用 cp437 → gbk 解码方案

#### 查询时

9. **先查知识库**：遇到被多次问及的问题类型时先查已有笔记和案例
10. **生成完成后逐条自查**：发现遗漏立即修正，不留给用户发现

## 六、文件关联速查表

| 从 | 到 | 关联方式 | 示例 |
|----|----|---------|------|
| 关键决策 | 项目产物 | 编号引用 | `决策依据：#D003` |
| 会议时间线 | 原始文档 | 双链 + 描述 | `[[原始纪要路径]]` |

| 复盘报告 | MEMORY/知识库 | 编号追溯 | `来源：XX项目复盘 YYYY-MM-DD` |
| 大知识库规范 | 项目文档 | 双链引用 | `[[../../01通用/工作流规范#章节]]` |
