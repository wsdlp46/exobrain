# 外脑核心工作流（Exobrain Workflow）

封装外脑（Exobrain）的五大核心工作流：搭结构、消化材料、辅助干活、复盘回流、自进化。

---

## 工作流 A：搭结构（Onboarding）

触发条件：用户首次使用，或说"帮我搭一套知识库结构"

### A1. 收集材料
要求用户提供至少 3 份工作文件作为样本。提示用户：
> 请提供至少 3 份最近的工作文件作为样本——可以是文档、会议纪要、方案、报告、设计稿，任何形式的工作产出物。

### A2. 特征识别
扫描所有文件，提取以下特征：

| 特征维度 | 判断方法 | 信号 |
|---------|---------|------|
| 文件类型 | 扩展名 + 内容 | 文档(.docx/.md)、表格(.xlsx)、演示(.pptx)、设计稿(.fig/.sketch)、PDF |
| 组织方式 | 文件路径结构 | 按项目分组 / 按业务线分组 / 按时间分组 / 扁平无结构 |
| 内容模式 | 关键词和结构 | 方案/提案 → 项目制；周报/月报 → 持续运营；学习笔记 → 个人 |
| 行业信号 | 术语和领域词 | 设计评审/组件库 → 设计；Sprint/Backlog → 开发；KPI/转化率 → 运营 |

### A3. 模式判断
根据特征识别结果，匹配目录模板：

| 判断条件 | 推荐模板 | 参考文件 |
|---------|---------|---------|
| 文件按项目文件夹组织 + 有交付物痕迹 | 项目制 | `references/directory-templates.md#模板1` |
| 按业务线/功能模块组织 + 时间线主轴 | 持续运营 | `references/directory-templates.md#模板2` |
| 两者兼有 | 混合模式 | `references/directory-templates.md#模板3` |
| 以个人笔记/学习材料为主 | 个人知识库 | `references/directory-templates.md#模板4` |

### A4. 推荐确认
向用户展示：
1. 推荐哪个模板及理由
2. 推荐的目录结构预览
3. 关键调整建议（如"建议增加01_设计规范目录"）

示例输出：
```markdown
## 推荐结构：项目制

**理由：** 你的 8 个文件中，6 个按项目文件夹组织（A项目/B项目/C项目），
且包含交付物（最终方案_v3.docx），符合项目制特征。

**推荐目录：**
├── _index.md
├── 00_通用/
├── 01_进行中/
├── 02_已完成/
├── 03_案例库/
├── 04_参考资料/
└── 05_关键决策.md

**调整建议：** 检测到 2 份设计规范文件，建议 `00_通用/` 下增加 `设计规范.md`。

是否按此方案创建？确认后我将创建目录结构和索引文件。
```

### A5. 创建执行
用户确认后：
1. 创建目录结构
2. 按 `references/knowledge-base-ops.md` 第二章附带的 Checklist 完成初始化：
   - 创建各目录 `_index.md`（含目录导航、文件清单）
   - 初始化 `01-项目输入/关键决策.md`（空文件，#D001 起编）
   - 初始化 `01-项目输入/会议纪要_时间线.md`（空文件）
   - 建立 `.workbuddy/memory/MEMORY.md`（设计约定）
3. 提示用户："结构已就绪，接下来可以逐步入库材料，说'入库这个文件'即可开始"

## 工作流 B：消化材料（Intake）

触发条件：用户说"入库"、"归档"、或直接提供文件路径/粘贴内容。

**委托给 `project-intake` Skill 执行。** 本工作流只做前置判断：
1. 确认用户当前的知识库目录已初始化
2. 确认文件所属项目/业务域（如不明确则询问）
3. 调用 `project-intake`
4. 入库完成后提示用户材料已分类至何处

---

## 工作流 C：辅助干活（Context Loading）

触发条件：用户说"开始干活"、"加载上下文"、或准备处理具体工作任务。

### C1. 路径确认
读取 `_index.md`，确认知识库结构有效。

### C2. 任务理解
询问用户当前要做什么（或从对话中推断），明确：
- 任务类型（写方案/做设计/写代码/做报告……）
- 所属项目/业务域
- 需要什么类型的参考材料

### C3. 分层检索
按优先级检索：

**第一层：关键决策**
- 读取关键决策文件（全局 + 当前项目）
- 筛选和当前任务相关的决策
- 格式：`D001：2024-03-15 确定使用 Vue3 替代 React —— 原因是团队技术栈一致性`

**第二层：历史案例**
- 在案例库中搜索相似案例
- 匹配维度：任务类型 > 行业领域 > 项目规模
- 返回最相关的 2-3 个案例摘要

**第三层：规范文件**
- 加载设计规范、模板、检查清单
- 只加载对标当前任务类型的规范

### C4. 上下文返回
以结构化摘要返回，不输出原始材料全文：

```markdown
## 上下文摘要

### 相关决策（3条）
- D001：...
- D005：...
- D012：...

### 相似案例（2个）
1. A项目方案 —— 相似度 80%，关键经验：...
2. B项目方案 —— 相似度 65%，关键经验：...

### 可参考规范
- 方案模板.md
- 设计规范.md——配色和组件部分
```

## 工作流 D：复盘回流（Retrospective）

触发条件：用户说"复盘"、项目结束、或完成重大里程碑。

**委托给 `project-retrospective` Skill 执行。** 本工作流做前置和后置处理：

### 前置
1. 确认复盘范围（单个项目/全局）
2. 收集复盘所需的上下文（项目文件、沟通记录、产出物）

### 后置
1. 复盘完成后，更新 `_index.md` 的项目状态
2. 将复盘产出的案例更新到案例库
3. 更新 `MEMORY.md` 中的工作模式记录

---

## 工作流 E：自进化

每次完成一个完整的项目周期后，更新 `MEMORY.md`：

### 应记录的内容
- 用户的行业和领域（从材料中的术语推断）
- 常用文档类型和模板偏好
- 跨项目的共性决策模式（如"多次在技术选型中优先考虑团队熟悉度"）
- 用户的输出风格偏好（正式/简洁/数据驱动等）

### 不应记录的内容
- 私密信息、敏感数据
- 纯技术实现细节
- 单次偶发的临时决策

### 进化效果
- 下次 Onboarding 时推荐更精准的目录结构
- 上下文加载时案例匹配更准
- 复盘时能区分偶发问题和模式性问题
